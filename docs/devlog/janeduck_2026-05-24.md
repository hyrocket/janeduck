### 단어 TTS 기능 설계 및 구현

#### 설계 검토 및 확정

TTS 기능 전체 스펙을 검토하고 아래 사항을 확정함.

**아키텍처 결정:**
- **캐시 전략**: Lazy 생성 (첫 재생 시점) + 영구 캐시 (한 번 만든 mp3 재생성 없음)
- **카드 fetch 방식**: A안 채택 — 카드 로딩 시 `word_audio` LEFT JOIN으로 `audio_url` 함께 전달
  - 이유: 세션당 수십 번 트리거되는 동작이라 매 클릭마다 API RTT 누적 부담 큼. LEFT JOIN은 PK라 비용 거의 0. 클라이언트 분기도 단순해짐 (audio_url 있으면 재생, NULL이면 API 호출)
  - 미캐시 단어는 POST /api/tts/word 호출 후 클라이언트 state에 결과 저장 (같은 세션 재호출 방지)
- **TTS 서비스**: Google Cloud Text-to-Speech (GA) — Gemini TTS (preview) 검토 후 기각. 이유: preview 모델은 상용 라이선스 불확실, 가격 변동 가능, SLA 없음
- **Voice**: `en-GB-Chirp3-HD-Laomedeia` (British English, 여성)
- **인증**: Service Account JSON → Base64 인코딩 후 env var 저장
- **저장**: Vercel Blob (public), 경로 `words/{encodeURIComponent(word)}.mp3`
- **Race condition**: INSERT ON CONFLICT DO NOTHING + SELECT fallback

**스코프:**
- 이번 라운드: 단어 TTS만 (공유 라이브러리 포함 전체 무료)
